# i.MX6U 启动与内存布局

---

## 一、 启动流程与 IVT 结构
在嵌入式 SoC（如 i.MX6U）上电后，处理器首先运行内部固化的 **Boot ROM** 程序。该程序负责寻找并解析“启动说明书”，即 **IVT (Image Vector Table)**。

### 1. IVT 核心字段解析
| 字段名称 | 功能描述 | 关键意义 |
| :--- | :--- | :--- |
| **Header** | 包含 Tag (0xD1) 和版本信息 | 用于 Boot ROM 校验数据合法性。 |
| **Entry** | 程序执行的起始地址 | 芯片初始化完成后，CPU 跳转执行的第一条指令位置。 |
| **DCD** | **Device Configuration Data** 的物理地址 | **核心概念**：由于裸机程序通常运行在 DDR 内存中，而 DDR 上电不可用，DCD 包含了一系列寄存器配置，用于初始化 DDR 控制器。 |
| **Self** | IVT 自身的物理起始地址 | 用于 Boot ROM 内部寻址参考。 |

## 二、 物理存储偏移 (Offset) 的原理
在 SD/EMMC 介质中，启动镜像（如 `load.imx`）并不是从 0 字节开始烧写的，而是存在 **1KB (1024 字节)** 的偏移。

### 1. 避开 MBR 分区表
* **MBR (Master Boot Record)**：传统磁盘分区表固定占用存储介质的前 **512 字节**。
* **兼容性**：SoC 厂商规定偏移 1KB，是为了防止烧写的启动镜像覆盖分区表，确保 SD 卡插入电脑后仍能正确识别分区空间。

### 2. 启动位置总结
* **SD/EMMC**：偏移 1KB。
* **Nand Flash / QSPI Flash**：通常偏移为 0。
* **计算公式**：
    $$实际烧写起始地址 = 存储介质起始地址 + 1024 \text{ Bytes}$$

## 三、 链接地址与链接脚本 (.lds)
链接脚本决定了程序在内存中的 **逻辑布局**。其核心任务是确保指令中涉及的跳转地址、变量地址与运行时的物理地址一致。

### 1. 链接地址偏移计算
如果在程序前方添加了 IVT、DCD 等头部信息（总计约 $3KB$），链接地址必须做出相应调整：
* 假设 DDR 起始地址为 $0x87800000$。
* 如果程序逻辑（`.bin` 部分）由于头部填充导致物理位置向后偏移了 $3KB$，则链接脚本中的起始地址必须设为 $0x87800000 + 3KB = 0x87800C00$。
* **后果**：若链接地址与实际加载地址不匹配，程序执行涉及绝对地址跳转（如全局变量访问、长跳转）时会指向错误的内存区域，导致“跑飞”。

## 四、 进阶：向 Linux 驱动开发过渡的知识储备

### 1. 必然保留的技能
* **地址空间意识**：Linux 驱动开发虽在虚拟地址空间操作，但涉及 DMA 或寄存器映射（I/O Mapping）时，仍需直接处理物理地址。
* **Makefile**：Linux 内核采用 **Kbuild** 编译系统，其本质是复杂的 Makefile 嵌套。掌握规则编写是开发驱动的前提。
* **链接逻辑**：虽然驱动模块（`.ko`）由内核自动加载，但了解 U-Boot 如何将 Linux 内核加载到 DDR 的指定偏移位置，是系统移植的核心。

### 2. 将被简化的部分
* **手写链接脚本**：在 Linux 应用和常规驱动开发中，编译器和内核会自动处理链接逻辑，极少需要开发者手动编写 `.lds`。
* **IVT/DCD 手动拼装**：在 Linux 开发阶段，通常使用厂商提供的工具（如 `mkimage`）自动生成包含头信息的 `u-boot.imx`，无需再用 WinHex 手动查看字节位。

### 3. Linux 开发的核心转向
* **设备树 (Device Tree)**：Linux 不再像裸机那样在代码里死磕寄存器地址，而是通过 `.dts` 文件描述硬件资源。
* **内核 API 框架**：学习如何使用内核提供的标准接口（字符设备、中断处理、并发锁）而非直接操控底层硬件位。

## 五、 关键术语对照表
* **Load Address (加载地址)**：镜像被拷贝到内存中的物理位置。
* **Execution Address (链接地址/运行地址)**：程序设计时预期运行的地址。
* **Bare Metal (裸机)**：无操作系统环境，代码直接控制硬件。
* **MBR (主引导记录)**：老式分区表标准，SD 卡启动时的“禁区”。
